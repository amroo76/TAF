Usage: TAF command-line options and associated properties.

  perl taf.pl [<options>]
  perl taf.pl --properties-file=/path/user_properties_file.properties
  perl taf.pl [<options>] --properties-file=/path/user_properties_file.properties

Notes:
* Command-line options override user and default properties files.
* Test Suite properties may be overridden by including the properties
  in your user properties file.
* TAF's defaults file:
      ./properties/default/taf_default.properties
  It is strongly recommended to always use user properties files to override.
* TAF runs off of action, command-line, and user properties files.

Note: generate-reports
   To regenerate report(s), copy the sub-results directories back into the
   results directory. Then run the command line again with:
       --action=generate-reports

Note: setup-client
   Some test suites (for example, sysbench-lua) require client binaries to be
   built against the active database software before they can be executed.
   Any action that includes "setup-client" will automatically invoke the test
   suite's client build routine using the currently active database install.

   In other cases (for example, hammerdbcli for the HammerDB test suite),
   no compilation is required, but the test suite still depends on specific
   client executables being present. In these cases, "setup-client" ensures
   that the required client executable exists and is available for use before
   test case execution begins.

   O = --database-iteration-mode={preserve|restart|reset|restore}
   P = taf.database_iteration_mode={preserve|restart|reset|restore}
                   : Controls what happens to the database between iterations.
                   :
                   : preserve
                   :     Do not modify the database contents between iterations.
                   :     The backend server is left running unless test_setup_mode
                   :     requires a drop/load. MainTestSetup runs only when
                   :     dictated by test_setup_mode.
                   :
                   : restart
                   :     Restart the backend server before each iteration. This
                   :     resets engine state (buffer pool, caches, background
                   :     threads) but preserves the database contents. After
                   :     restart, MainTestSetup runs only if required by
                   :     test_setup_mode.
                   :
                   : reset
                   :     Fully recreate the database before each iteration. This
                   :     performs a backend stop, DbInit(), and backend start,
                   :     producing a cold backend state *and* fresh database
                   :     contents. MainTestSetup is always invoked after reset,
                   :     regardless of test_setup_mode.
                   :
                   : restore
                   :     Use a saved database snapshot to restore the datadir
                   :     before each iteration. On the first iteration, a full
                   :     MainTestSetup is performed. After setup completes, the
                   :     framework creates a restore image (tar.gz) in the
                   :     directory specified by database_restore_image_dir.
                   :     On subsequent iterations, the backend is stopped,
                   :     the datadir is replaced with the restore image, and
                   :     the backend is started cold. MainTestSetup is skipped
                   :     after the first iteration. Warmup is always required.
                   :
                   : Note:  reset and restore override test_setup_mode and will
                   :        perform test setup on the first iteration.
                   : Default(preserve)

   O = --database-restore-image-dir=DIR

{TEST SETUP MODES}
----------------

   O = --test-setup-mode=<mode>
   P = taf.test_setup_mode=<mode>
                   : Default(once)

   The framework supports four mutually exclusive modes for running MainTestSetup. 
   
   These are controlled by the option:

    test_setup_mode = skip | once | per_test | per_iter

    Meaning:

      skip      : Never run MainTestSetup.
      once      : Run MainTestSetup once for the entire run.
      per_test  : Run MainTestSetup once for each test case name.
      per_iter  : Run MainTestSetup on every iteration.

      Examples:

        test_setup_mode = skip
        test_setup_mode = once
        test_setup_mode = per_test
        test_setup_mode = per_iter

{Database Software}

Note: Active Install Marker
   TAF maintains a small marker file that records which database software
   installation is currently considered "active". This marker allows TAF to
   consistently determine which installation should be used for actions such as
   setup-client, setup-db, start-db, and run-tests.

   The active install marker is updated automatically when:
       - A new installation is performed using --db-software-install-packages
       - The user explicitly selects an install using:
             --set-active-database-software-install
       - The user specifies an existing install using:
             --db-software-install-dir=<path>
         (Selecting an install directory implicitly makes it the active install.)

   The marker is stored in the TAF working directory as a small text file
   (for example, .taf-active-install.txt) containing the name of the active
   installation directory.

   This allows TAF to provide the following capabilities:

       - Test suites can reliably locate the correct database binaries.
       - Client build routines (for example, sysbench-lua) can compile against
         the correct installation without requiring user input.
       - Suites that depend on pre-existing executables (for example, hammerdbcli
         for the HammerDB test suite) can verify that the required client
         executable exists under the active install.
       - Actions such as setup-client, setup-db, start-db, and run-tests always
         operate against the same, explicitly selected installation.

   To view the currently active installation, use:
       --list-active-db-install

   To change the active installation, use:
       --set-active-database-software-install

{DB Process Rest Watch}

   O = --db-process-rest-enable
   P = taf.db_process_rest_enable=<bool>

                   : process. When enabled, TAF waits for the DB process
                   : to reach a stable rest state before continuing.
                   : Default(false)

   O = --db-process-rest-low=<percent>
   P = taf.db_process_rest_low=<percent>
                   : CPU percent considered "at rest". Samples at or
                   : below this value count toward the required number
                   : of consecutive rest samples.
                   : Default(5)

   O = --db-process-rest-high=<percent>
   P = taf.db_process_rest_high=<percent>
                   : CPU percent that forces a reset of the rest counter.
                   : Samples at or above this value indicate the process
                   : is still active.
                   : Default(20)

   O = --db-process-rest-consecutive=<count>
   P = taf.db_process_rest_consecutive=<count>
                   : Number of consecutive "at rest" samples required
                   : before the DB process is considered fully idle.
                   : Default(10)

   O = --db-process-rest-max-attempts=<count>
   P = taf.db_process_rest_max_attempts=<count>
                   : Maximum number of sampling cycles before giving up
                   : and reporting that the DB process did not reach a
                   : rest state.
                   : Default(200)

   O = --db-process-rest-interval=<seconds>
   P = taf.db_process_rest_interval=<seconds>
                   : Seconds to sleep between sampling cycles. This is
                   : in addition to the 1-second CPU sampling window.
                   : Default(2)

   O = --db-process-rest-verbose
   P = taf.db_process_rest_verbose=<bool>
                   : Enable verbose diagnostic snapshots during rest
                   : detection. Intended for troubleshooting only.
                   : Default(false)
